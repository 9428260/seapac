# Skills — 현실 세계와 비교한 동작 설명

각 스킬이 **현실의 전력·요금·ESS 운영**과 어떻게 대응하는지 정리합니다.

---

## 1. EnergyForecastSkill

### 1.1 현실 세계에서의 “에너지 예측”

- **전력 수요 예측(Load forecast)**  
  전력회사·에너지 커뮤니티는 **수요(부하) 곡선**을 미리 예측해 발전·수급 계획을 세웁니다.  
  - 날씨(기온·습도), 요일·휴일, 과거 수요 패턴, 이벤트 등을 반영합니다.  
  - 단위는 보통 kW·MW, 15분·1시간 해상도입니다.

- **태양광(PV) 발전량 예측**  
  일사량·운량·설비용량을 바탕으로 **발전량(kW)**을 예측합니다.  
  - 기상 예보·일사 센서·역사 데이터를 사용합니다.

- **모델 선택**  
  현장에서는 데이터 양·품질·해상도에 따라 **통계 모델(회귀, 시계열)** 또는 **ML(LightGBM, XGBoost, LSTM 등)**을 선택합니다.  
  - 데이터가 충분하면 복잡한 모델, 적으면 단순 모델을 쓰는 경우가 많습니다.

- **예측 정확도 평가**  
  실제 계측값과 예측값을 비교해 **MAPE(평균 절대 백분율 오차)**, **RMSE** 등으로 평가합니다.  
  - 전력계에서는 MAPE 5~10% 이내를 목표로 두는 경우가 많습니다.

### 1.2 스킬 동작과의 대응

| 스킬 기능 | 현실 세계에서의 의미 | 코드에서의 동작 |
|-----------|----------------------|-----------------|
| **select_model(n_samples, prosumer_type)** | “데이터가 얼마나 있는지, 어떤 수용가 유형인지에 따라 예측 모델을 고르는” 절차 | 샘플 수 ≥ 5,000이면 LightGBM, 아니면 XGBoost 선택. 프로슈머 타입은 현재 로직에서는 분기 없음(추가 확장용). |
| **evaluate_forecast(actual, predicted)** | “예측이 실제와 얼마나 맞는지” 수치로 평가 | MAPE(실제값 > 1 kW인 구간만), RMSE 계산. 현실의 성능 리포트와 동일한 지표. |
| **build_forecast_result(...)** | “어느 시각에 실제값·예측값이 얼마였는지” 정리한 시계열 테이블 | timestamp, 실제값 컬럼, 예측값 컬럼으로 DataFrame 생성. 현실의 **예측 vs 실측 비교 시트**에 해당. |

**요약**:  
- **select_model** → 현실의 “데이터 규모·유형에 맞는 모델 선택”을 단순 규칙으로 구현.  
- **evaluate_forecast** → 현실의 **예측 정확도 평가(MAPE/RMSE)**와 동일한 역할.  
- **build_forecast_result** → 현실의 **시간별 예측·실측 기록표**를 DataFrame으로 만드는 것.

## 2. TariffAnalysisSkill

### 2.1 현실 세계에서의 “전기 요금·TOU”

- **TOU(Time-of-Use) 요금**  
  시간대에 따라 **단가가 달라지는** 요금제입니다.  
  - **한국**: 산업용 등에서 심야(off-peak)·준최대·최대(on-peak) 구간으로 나누어 단가를 다르게 책정합니다.  
  - **해외**: 주간 피크·오프피크 2~3구간이 흔합니다.

- **실제 구간 예시(한국 산업용 등)**  
  - **심야(off-peak)**: 23~09시 등 — 단가 낮음.  
  - **준최대(mid)**: 09~12, 13~18, 21~23시 등 — 기본 단가.  
  - **최대(on-peak)**: 12~13, 18~21시 등 — 단가 높음.

- **요금 절감 시뮬레이션**  
  “ESS를 쓰지 않았을 때 전력비”와 “ESS 충방전을 적용했을 때 전력비”를 비교해 **절감액·절감률**을 계산합니다.  
  - 현실에서는 계약 전력·부담금·연료비조정액 등이 더해지지만, 여기서는 **사용량 × 단가**만으로 단순화합니다.

### 2.2 스킬 동작과의 대응

| 스킬 기능 | 현실 세계에서의 의미 | 코드에서의 동작 |
|-----------|----------------------|-----------------|
| **TOU_PERIODS** | “몇 시부터 몇 시를 심야/준최대/최대로 볼 것인가” + 구간별 단가 배율 | 0~9, 23~24시 = off_peak(0.7배), 12~13·18~21시 = on_peak(1.5배), 나머지 = mid_peak(1.0배). 한국 산업용 TOU를 **간략화**한 형태. |
| **classify_period(hour)** | “이 시간은 어떤 요금 구간인가?” 판별 | hour로 off_peak / mid_peak / on_peak 반환. 현실의 **요금 구간 판정**과 동일. |
| **analyze(df)** | “각 시각의 부하·단가·구간·추정 전력비”를 한 테이블로 정리 | timestamp → hour → tou_period → tou_multiplier, estimated_cost = load_kw × price_buy × 0.25(15분). 현실의 **시간별 요금 분석표**에 해당. |
| **cost_saving_simulation(df, ess_schedule)** | “ESS 없이 쓴 비용” vs “ESS 충방전 반영 후 비용”으로 절감액 계산 | 기준 비용 = Σ(load_kw × price_buy × 0.25). ESS 방전 시 그 구간의 grid 구매량 감소, 충전 시 증가로 adjusted_load 계산 후 동일 단가로 비용 합산. **실제 요금서 비교 시뮬레이션**과 같은 논리. |
| **summarize_by_period(df)** | “구간별로 부하·요금을 집계” | off_peak / mid_peak / on_peak별 평균 부하·평균 단가·총 비용·스텝 수. 현실의 **구간별 요금·수요 리포트**에 해당. |

**요약**:  
- **TOU 구간**은 한국 산업용 TOU를 단순화한 것이고, **절감 시뮬레이션**은 “계통에서 사서 쓰는 양 × 단가”만으로 비용을 계산하는 **현실의 요금 비교·시뮬레이션**을 구현한 것입니다.  
- 실제 요금서에는 기본요금·부담금·세금 등이 더 있으므로, 스킬은 **사용량·단가 기반의 상대적 비교**용으로 보는 것이 맞습니다.

## 3. ESSOptimizationSkill

### 3.1 현실 세계에서의 “ESS 운영”

- **ESS(Energy Storage System)**  
  전기를 **저장했다가 필요한 시간에 방전**하는 장치입니다.  
  - **용량(kWh)**: 저장할 수 있는 에너지 양.  
  - **출력(kW)**: 한 순간에 넣거나 뺄 수 있는 전력 크기.  
  - **SOC(State of Charge)**: 현재 저장된 양을 용량 대비 % 또는 kWh로 나타낸 값.

- **Peak Shaving(피크 삭감)**  
  **최대 부하(피크)**가 넘지 않도록, 피크 시간대에 ESS를 **방전**해 계통에서 받는 전력을 줄입니다.  
  - 현실: 계약 전력·최대 수요 관리·수요 반응(DR) 비용 절감 목적.  
  - “피크 한도”를 정해 두고, 그 위로 나가면 방전으로 보충하는 방식입니다.

- **Valley Filling**  
  부하가 **낮은 시간대**에 계통 전력을 받아 ESS를 **충전**해 두는 것입니다.  
  - 심야 전기 요금이 쌀 때 충전하는 것이 현실에서 흔한 사례입니다.

- **TOU 연동 운영**  
  **단가가 높은 시간**에는 방전해 구매량을 줄이고, **단가가 낮은 시간**에는 충전합니다.  
  - 현실의 **요금 절감형 ESS 운영**과 같은 개념입니다.

- **운영 제약**  
  - **SOC 상·하한**: 과방전·과충전 방지(예: 20~90%).  
  - **출력 한도(kW)**: 인버터·PCS 용량 제한.  
  - **시간 간격(dt)**: 15분 단위면 0.25시간당 충방전량 = power_kw × 0.25 (kWh).

### 3.2 스킬 동작과의 대응

| 스킬 기능 | 현실 세계에서의 의미 | 코드에서의 동작 |
|-----------|----------------------|-----------------|
| **__init__(bess_kwh_cap, bess_kw_cap, min_soc_pct, max_soc_pct, ...)** | “어떤 규모의 ESS를, 어떤 SOC 범위로 운영할 것인가” 설정 | 용량(kWh)·출력(kW)·SOC 최소/최대 비율·초기 SOC·시간 간격(dt_h=0.25 → 15분)을 저장. 현실의 **ESS 사양·운영 제약** 설정. |
| **peak_shaving_schedule(net_load, timestamps, peak_limit_kw)** | “피크 한도를 넘을 때만 방전, 한가할 때 충전”하는 **피크 삭감 스케줄** | net_load > peak_limit 이고 SOC 여유 있으면 → 방전(필요량·가용량·출력 한도 중 최소). net_load < peak_limit×0.5 이고 SOC 여유 있으면 → 충전. 그 외 대기. 15분마다 SOC 갱신. **현실의 피크 삭감 로직**을 단순화한 것. |
| **tou_schedule(net_load, timestamps, price_series)** | “비쌀 때 방전, 쌀 때 충전”하는 **TOU 연동 스케줄** | 가격 상위 70% 이상이면 방전(가능한 만큼), 하위 30% 이하면 충전. **현실의 요금 기반 충방전**과 같은 전략. |
| **summarize(schedule_df)** | “이번 스케줄로 얼마나 충·방전했는지” 요약 | 충전/방전/대기 스텝 수, 총 충전·방전량(kWh), SOC 최소·최대. 현실의 **ESS 운영 리포트**와 동일한 항목. |

**요약**:  
- **peak_shaving_schedule** → 현실의 **피크 한도 관리·피크 삭감**을, 한도 초과 시 방전·한가 시 충전 규칙으로 구현.  
- **tou_schedule** → 현실의 **요금 기반 충방전(비쌀 때 방전, 쌀 때 충전)**을 가격 분위수로 구간 나눠 구현.  
- **SOC·출력 한도·dt** → 현실의 **배터리 사양·운영 제약**을 파라미터로 받아 반영합니다.

## 4. 현실 세계와의 차이·한계 (공통)

| 구분 | 현실 | 스킬 |
|------|------|------|
| **데이터** | 계측·시장가·실제 설비 제어 | 시뮬레이션·예측 결과·합성 데이터 |
| **요금** | 기본요금·부담금·세·복잡한 TOU | 사용량×단가·간단한 TOU 구간만 |
| **ESS** | 효율·열화·PCS 제어 지연 | 효율 100% 가정, 즉시 충방전 |
| **해상도** | 15분·1시간·실시간 계약에 따라 상이 | 15분(dt_h=0.25) 고정 |

스킬은 **현실의 개념·지표·운영 전략**을 **단순화·규칙화**한 것이므로, 실제 계통·요금·ESS 제어와 정량적으로 맞추려면 파라미터와 로직을 현장 데이터에 맞게 조정해야 합니다.
