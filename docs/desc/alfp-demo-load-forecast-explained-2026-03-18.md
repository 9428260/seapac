# ALFP 데모 데이터로 보는 전력사용량 예측 설명 (2026-03-18)

## 1. 먼저, ALFP 데모 데이터는 무엇인가

ALFP가 기본으로 읽는 데모 학습 데이터는 `data/train_2026_seoul.pkl`이다. 실제 확인 결과 이 데이터는 다음 성격을 가진다.

- 기간: 2026-01-01 00:00:00+09:00 ~ 2026-03-06 07:45:00+09:00
- 해상도: 15분 단위
- 프로슈머 수: 20
- 전체 레코드 수: 123,520
- 프로슈머 타입: `Commercial`, `EnergyHub`, `Industrial`, `Residential`, `Rural`

각 시점마다 대략 이런 값이 들어 있다.

- `load_kw`: 실제 전력 사용량
- `pv_kw`: 태양광 발전량
- `wt_kw`: 풍력 발전량
- `bess_soc_kwh`: 배터리 저장 상태
- `bess_ref_power_kw`: 배터리 기준 출력
- `controllable_load_kw`: 제어 가능한 부하
- `price_buy`, `price_sell`, `price_p2p`: 전력 가격

이 데이터는 완전한 실측 원본이라기보다, ELIA 기반 시계열과 IEEE141 계통 정보를 조합해 만든 재현용 데이터다. 즉 "현실을 흉내 낸 데모 운영 데이터"라고 보는 것이 정확하다.

## 2. 현실 세계에 비유하면, ALFP는 무엇을 하는 시스템인가

현실에 비유하면 ALFP는 대형 건물이나 공장 단지의 에너지 운영실에 있는 "예측 담당 팀장"과 비슷하다.

이 팀장은 매 15분마다 다음을 본다.

- 지금까지 전기를 얼마나 썼는지
- 지금 태양광이 얼마나 발전했는지
- 요금이 비싼 시간인지
- 날씨가 어떤지
- 평일인지 주말인지
- 어제 같은 시간, 지난주 같은 시간에 얼마나 썼는지

그리고 이렇게 묻는다.

"오늘 오후 3시쯤 우리 건물은 전기를 얼마나 쓸까?"

ALFP의 전력사용량 예측은 바로 이 질문에 답하는 과정이다.

## 3. 전체 흐름을 한 줄로 요약하면

ALFP의 부하 예측은 다음 순서로 일어난다.

1. 데모 데이터에서 특정 프로슈머의 시계열을 꺼낸다.
2. 데이터가 깨졌는지 검사하고 보정한다.
3. 사람이 직관적으로 볼 힌트들을 feature로 만든다.
4. 어떤 모델을 쓸지 planner가 정한다.
5. 모델이 과거 구간으로 학습한다.
6. 뒤쪽 검증 구간에 대해 미래 사용량을 예측한다.
7. 실제값과 예측값을 비교해 성능을 평가한다.
8. 그 결과를 이후 PV 예측, Net Load 계산, 운영 의사결정으로 넘긴다.

## 4. 단계별 설명: 현실 비유와 코드 기준을 같이 보면

### 4.1 데이터 로드: "건물 관리대장이 지난 기록을 꺼내는 단계"

`alfp/data/loader.py`에서 `load_dataset()`이 pkl 파일을 읽고, `get_timeseries()`가 특정 `prosumer_id`의 이력을 뽑는다.

현실 비유로 보면, 관리실 직원이 "48번 상업용 건물의 1월부터 3월 초까지 15분 간격 전력 기록"을 엑셀에서 꺼내는 단계다.

여기서 중요한 점은 ALFP가 아무 프로슈머나 한 번에 다 예측하는 것이 아니라, 보통 특정 프로슈머 하나를 골라 그 대상의 시계열을 중심으로 학습한다는 점이다.

## 4.2 데이터 품질 검사: "센서 오작동이나 빈칸을 먼저 정리하는 단계"

`alfp/agents/data_quality.py`는 예측 전에 데이터를 정리한다.

실제로 하는 일은 다음과 같다.

- `load_kw`, `pv_kw`, 가격 컬럼의 결측값 탐지
- 시간 기반 보간
- `load_kw`, `pv_kw`의 이상치 IQR 클리핑
- 음수값 0 처리

현실에 비유하면 이런 상황이다.

- 어떤 15분 구간은 계량기 값이 비어 있다.
- 어떤 시점은 센서 튐 때문에 말도 안 되는 값이 들어갔다.
- 태양광 발전량이 음수로 찍혔다.

사람이 운영실에서 엉터리 값을 그대로 보고 판단하면 예측이 망가진다. 그래서 ALFP는 먼저 "측정 기록을 사람이 믿을 수 있는 상태"로 만든다.

## 4.3 Feature Engineering: "단순 사용량 숫자를 운영 맥락으로 번역하는 단계"

이 단계가 예측의 핵심 중 하나다. `alfp/agents/feature_engineering.py`가 수행한다.

모델은 `load_kw` 숫자만 보고 미래를 잘 맞히기 어렵다. 그래서 ALFP는 과거 기록을 바탕으로 이런 힌트를 만든다.

- 시간 정보: `hour`, `minute`, `weekday`, `month`
- 패턴 정보: 주말 여부, 공휴일 여부, 피크 시간 여부
- 계절성 정보: `hour_sin`, `hour_cos`, `month_sin`, `month_cos`
- 과거 사용량 힌트: `load_kw_lag1`, `load_kw_lag4`, `load_kw_lag96`, `load_kw_lag672`
- 과거 PV 힌트: `pv_kw_lag1`, `pv_kw_lag4`, `pv_kw_lag96`
- 이동평균/변동성: `load_kw_roll_mean_4`, `load_kw_roll_mean_96` 등
- 가격 힌트: `price_buy_lag4`, `price_buy_roll_mean_96`
- 배터리 상태 힌트
- 날씨 힌트: OpenWeather 기반 온도, 습도, 구름량, 풍속

현실에 비유하면, 숙련된 전력 운영자가 머릿속으로 이런 질문을 동시에 하는 것과 같다.

- 지금이 오전 9시인가 밤 11시인가?
- 평일인가 토요일인가?
- 어제 같은 시간에도 사용량이 높았나?
- 일주일 전 같은 시간도 비슷했나?
- 지금 날씨가 흐려서 태양광 기여가 줄어들 것 같은가?
- 전기요금이 비싸서 설비 운전 패턴이 달라질 가능성이 있는가?

즉 feature engineering은 "숫자를 문맥으로 바꾸는 작업"이다.

## 4.4 Forecast Planner: "어떤 예측 전략이 맞을지 선임 엔지니어가 먼저 판단하는 단계"

`alfp/agents/forecast_planner.py`는 예측을 바로 시작하지 않고, 먼저 데이터 성격을 해석한다.

planner는 이런 통계를 본다.

- 데이터 개수는 충분한가
- 프로슈머 타입은 무엇인가
- 부하 평균과 표준편차는 어떤가
- 부하 변동성은 높은가
- PV 영향은 큰가
- 요청 horizon은 몇 스텝인가
- 요금제가 시간대별로 많이 흔들리는가

그리고 후보 전략을 비교해 `lgbm` 또는 `xgboost` 같은 모델과 설정을 정한다.

현실 비유로 보면, 예측을 맡은 선임 엔지니어가 먼저 이렇게 판단하는 것이다.

"이 건물은 패턴이 비교적 안정적인 편이니 설명 가능성이 좋은 쪽이 낫겠다."
"이 대상은 태양광 비중이 커서 기상 변화에 민감하니 피크 대응이 중요하겠다."
"데이터가 충분하니 더 복잡한 트리 모델을 써도 되겠다."

즉 ALFP는 모델을 무조건 하나로 고정하지 않고, 데이터를 보고 전략을 고른다.

## 4.5 모델 선택: "어떤 예측기술자를 투입할지 정하는 단계"

`alfp/skills/energy_forecast.py`의 `select_model()`을 보면 기본 규칙이 단순하다.

- 샘플 수가 충분하면 `LightGBM`
- 아니면 기본값 `XGBoost`

현실에 비유하면 이렇다.

- 데이터가 충분하고 패턴이 복잡하면, 더 세밀한 분석가를 투입한다.
- 데이터가 적으면, 덜 과감하고 안정적인 분석가를 투입한다.

여기서 중요한 점은 ALFP가 LLM으로 사용량 숫자를 직접 뽑는 것이 아니라, 실제 숫자 예측은 `LightGBM` 또는 `XGBoost` 같은 전통 ML 모델이 한다는 점이다.

## 4.6 Load Forecast 학습: "과거 운영일지를 보고 사용 패턴을 익히는 단계"

`alfp/agents/load_forecast.py`에서 실제 부하 예측이 일어난다.

흐름은 다음과 같다.

1. `feature_df`와 `feature_names`를 받는다.
2. 시계열을 시간 순서대로 train/validation으로 나눈다.
3. 선택된 모델을 생성한다.
4. `load_kw`를 타깃으로 학습한다.
5. 검증 구간의 `predicted_load_kw`를 만든다.

여기서 분할 방식이 실무적으로 중요하다. 랜덤 샘플링이 아니라 "앞 구간으로 학습하고 뒤 구간을 검증"한다. 시계열 예측이기 때문이다.

현실 비유로 보면, 예측 담당자가 1월부터 지난주까지의 기록을 공부한 뒤, 이번 주 후반 운영을 맞혀보는 방식이다. 미래를 예측하는 문제인데 과거와 미래를 섞어버리면 현실성이 없기 때문이다.

### 짧은 데이터일 때 왜 반으로 나누는가

코드에는 1일 수준의 짧은 데이터일 때 validation을 후반부에 두는 예외가 있다. 낮과 오후 구간이 검증셋에 들어가도록 하기 위해서다.

현실에서는 태양광과 거래 판단이 주로 낮 시간에 중요하다. 검증셋이 새벽만 포함하면 "낮 피크와 태양광 영향"을 제대로 확인하지 못한다. ALFP는 이 점을 의식해 검증 구간을 잡는다.

## 4.7 PV 예측: "태양광이 얼마나 덜어줄지도 따로 계산하는 단계"

실제 운영에서는 "얼마나 쓸까"만 중요하지 않다. "그중 태양광이 얼마나 메워줄까"도 중요하다.

`alfp/agents/pv_forecast.py`는 별도 모델로 `pv_kw`를 예측한다. 여기서는 모든 feature를 다 쓰지 않고 PV에 더 관련된 feature만 추린다.

- 시간
- 계절
- 날씨
- PV 과거값
- 일부 가격 정보

현실 비유로 보면, 건물 전체 수요를 맞히는 담당자와 태양광 발전량을 맞히는 담당자가 따로 있는 셈이다. 둘을 분리해야 더 정확하다.

## 4.8 Net Load 계산: "결국 한전에 실제로 얼마나 더 받아와야 하는지 계산하는 단계"

`alfp/agents/net_load_forecast.py`는 다음 공식을 쓴다.

`predicted_net_load_kw = predicted_load_kw - predicted_pv_kw`

현실 비유로 보면 이 값이 가장 실무적이다.

- 건물이 100kW를 쓸 것 같고
- 태양광이 30kW를 만들어줄 것 같으면
- 외부 전력망에서 실제로 메워야 할 양은 70kW다

ALFP는 이 값을 보고 피크 시간, ESS 충방전, 거래 전략을 뒤에서 결정한다.

흥미로운 점은 예측 Net Load는 음수를 허용한다는 것이다. 즉 PV가 부하보다 크면 잉여 전력이 생긴다고 본다. 이는 뒤에서 거래 추천에 활용된다.

## 4.9 Validation: "예측이 실제와 얼마나 가까웠는지 사후 점검하는 단계"

`alfp/agents/validation.py`에서 성능을 계산한다.

대표 지표는 다음과 같다.

- `MAE`: 평균적으로 몇 kW 틀렸는가
- `RMSE`: 큰 오차를 더 민감하게 보는 오차
- `MAPE`: 실제 대비 몇 퍼센트 틀렸는가
- `Peak Error`: 최대 피크를 얼마나 잘 맞혔는가

현실 비유로 보면, 운영실 책임자가 다음을 체크하는 단계다.

- 평소 사용량은 대체로 맞았는가
- 큰 오차가 자주 났는가
- 가장 위험한 피크 시간은 제대로 맞혔는가

이 중 특히 피크 예측이 중요한 이유는, 전력 운영에서는 평균 오차보다 "가장 바쁜 시간"을 놓치는 것이 더 비싸고 위험할 수 있기 때문이다.

## 5. 이 예측은 현실에서 어떻게 이해하면 좋은가

ALFP의 전력사용량 예측은 점쟁이처럼 미래를 맞히는 것이 아니다. 더 정확한 비유는 "운영실의 숙련자가 과거 기록, 달력, 날씨, 요금표를 동시에 보며 다음 근무조 전력 수요를 추정하는 일"이다.

예를 들어 상업용 건물이라면 이런 식으로 작동한다고 이해하면 된다.

1. 오전 8시 이후 사람이 출근하면 부하가 올라간다.
2. 점심 이후 냉난방과 장비 부하가 바뀐다.
3. 주말이면 패턴이 달라진다.
4. 흐리면 PV가 줄어든다.
5. 전기요금이 비싼 시간대면 일부 설비 운전 패턴도 달라질 수 있다.
6. 어제 같은 시간과 지난주 같은 시간은 강한 힌트가 된다.

ALFP는 바로 이런 인간적 추론을 feature와 트리 모델로 기계화한 것이다.

## 6. 데모 데이터 기준으로 왜 이런 예측이 가능한가

이 데모 데이터는 15분 단위이고, 여러 타입의 프로슈머가 있으며, 부하 외에도 PV, 가격, 배터리 상태 같은 운영 맥락을 함께 담고 있다. 그래서 단순 회귀보다 훨씬 풍부한 패턴을 학습할 수 있다.

특히 다음 정보가 같이 있다는 점이 크다.

- 시간 축 패턴
- 과거 부하 패턴
- 태양광 패턴
- 가격 패턴
- 프로슈머 타입 차이
- 배터리/제어 가능 부하 맥락

현실 비유로 보면, 전력 수요만 적힌 장부가 아니라 "날씨 메모, 요금표, 설비 상태표"까지 같이 있는 운영일지에 가깝다.

## 7. 한 문장으로 정리하면

ALFP 데모 데이터 기반 전력사용량 예측은 "특정 프로슈머의 15분 단위 과거 전력 사용 기록에 시간, 날씨, 가격, 태양광, 배터리 맥락을 덧붙여 특징을 만들고, 그 특징으로 LightGBM 또는 XGBoost를 학습시켜 다음 구간의 `load_kw`를 맞히는 과정"이다.

## 8. 실무 감각으로 다시 정리

사람이 운영실에서 하는 일을 기준으로 보면 다음과 같다.

- 데이터 로드: 지난 운영일지 펼치기
- 데이터 품질 검사: 계량기 오류와 빈칸 정리
- feature engineering: 숫자를 상황 정보로 번역
- forecast planner: 어떤 예측 전략이 맞을지 결정
- load forecast: 실제 전력사용량 예측
- pv forecast: 태양광 보정치 예측
- net load: 외부 전력망에서 실제로 필요한 양 계산
- validation: 예측이 실제 운영과 얼마나 맞았는지 검수

이렇게 보면 ALFP의 예측은 단순 모델 호출이 아니라, 현실 운영팀의 판단 흐름을 코드로 옮긴 절차라고 이해하면 된다.
