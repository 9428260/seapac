# Product Requirement Document (PRD)

## SEAPAC Final Parallel Execution Agents

### Policy Management Agent · Eco Saver Agent · Storage Management Agent

------------------------------------------------------------------------

# 1. Overview

This document defines the Product Requirement for the **Final Parallel
Execution Layer** of the SEAPAC (Smart Energy A2A Platform for Apartment
Community) multi‑agent energy management system.

The new architecture introduces three specialized agents that operate
**in parallel before final execution**:

-   Policy Management Agent
-   Eco Saver Agent
-   Storage Management Agent (PV + ESS)

These agents evaluate candidate actions generated by the energy market
negotiation stage and ensure that:

-   policies and regulations are satisfied
-   device constraints are respected
-   user energy savings are optimized
-   only safe and feasible actions are executed

The output is merged by the **Execution Orchestrator**, which approves
final actions to be applied to the energy system.

------------------------------------------------------------------------

# 2. Objectives

## 2.1 Primary Goals

1.  Ensure **policy compliance** before executing energy market
    decisions.
2.  Guarantee **physical feasibility** of PV and ESS device operations.
3.  Provide **personalized energy saving recommendations** to residents.
4.  Reduce **decision latency** through parallel agent execution.
5.  Improve **system explainability and auditability**.

## 2.2 Business Impact

Expected improvements:

-   Increased internal energy trading revenue
-   Reduced peak demand cost
-   Improved ESS lifecycle management
-   Higher user engagement in energy savings
-   Transparent and auditable system operation

------------------------------------------------------------------------

# 3. System Context

The new agents operate in the final stage of the SEAPAC decision
pipeline.

## System Workflow

1.  State Collection
2.  State Translation
3.  Multi‑Agent Negotiation
4.  **Parallel Evaluation Layer (New)**
5.  Execution Orchestrator
6.  Device / Market Execution
7.  KPI Evaluation

------------------------------------------------------------------------

# 4. Architecture

## Final Parallel Execution Layer

Three agents run simultaneously.

Policy Management Agent Eco Saver Agent Storage Management Agent

Each agent receives the same candidate action bundle and returns
evaluation results.

The Execution Orchestrator merges the responses and produces final
executable actions.

------------------------------------------------------------------------

# 5. Agent Definitions

# 5.1 Policy Management Agent

## Purpose

Validate candidate actions against regulatory policies, safety rules,
and operational constraints.

## Responsibilities

-   Validate market rules
-   Validate trading price limits
-   Validate trading volume limits
-   Verify ESS safety constraints
-   Validate PV export limits
-   Enforce resident consent policies
-   Apply operational priority rules

## Inputs

-   Candidate action bundle
-   Market policy configuration
-   Device safety constraints
-   Regulatory requirements
-   Resident preference data

## Outputs

-   Approved actions
-   Rejected actions
-   Modified actions
-   Policy violation report
-   Risk score

## Key Modules

-   Policy Rule Engine
-   Constraint Validator
-   Exception Handler
-   Audit Logger

------------------------------------------------------------------------

# 5.2 Eco Saver Agent

## Purpose

Generate personalized energy‑saving recommendations for residents
without disrupting market operations.

## Responsibilities

-   Analyze household consumption patterns
-   Detect peak demand opportunities
-   Generate behavioral recommendations
-   Estimate potential savings
-   Prevent notification fatigue

## Inputs

-   Load forecast
-   Household segmentation
-   Pricing forecast
-   Candidate system actions
-   User preference data

## Outputs

-   Energy saving recommendations
-   Estimated savings
-   Acceptance probability
-   Notification message payload

## Example Recommendation

"Running the washing machine after 21:00 may save approximately 500
KRW."

------------------------------------------------------------------------

ESS Operation Manager

## ESS Operation Manager

### Responsibilities

-   Enforce charge/discharge limits

-   ESS charge schedule
-   ESS discharge schedule

-   Battery degradation estimate

------------------------------------------------------------------------

# 6. Parallel Execution Model

All agents run asynchronously.

Each receives identical input data.

The Execution Orchestrator waits for responses and merges results.

## Agent Priority

1.  Policy Agent
2.  Storage Agent
3.  Market Decision
4.  Eco Saver Agent

Policy and Storage agents have veto authority.

Eco Saver acts as advisory.

------------------------------------------------------------------------

# 7. Execution Orchestrator

The Execution Orchestrator merges results from all parallel agents.

## Responsibilities

-   Merge agent outputs
-   Resolve conflicts
-   Apply policy veto rules
-   Ensure device feasibility
-   Generate final executable action bundle

## Output Types

Market Actions

-   energy sell orders
-   energy buy orders

Device Actions

-   ESS charging
-   ESS discharging
-   PV output control

Advisory Actions

-   energy saving recommendations
-   resident notifications

# 9. Failure Handling

## Policy Agent Failure

System enters safe mode and blocks execution.

## Storage Agent Failure

Device control is disabled.

## Eco Saver Failure

System continues without recommendations.
